# Project Overview

This repository automates the deployment of a ProShop MERN application on AWS using Terraform and Ansible.

The project provisions two EC2 instances:

- `App Server`: runs Nginx, the React frontend, the Node.js backend, MongoDB, Node Exporter, Promtail, and Fail2ban
- `Monitoring Server`: runs Prometheus, Grafana, Loki, Alertmanager, and Fail2ban

The goal is to provide a compact DevOps project that covers infrastructure provisioning, configuration management, HTTPS automation, monitoring, logging, and alerting without introducing Kubernetes or multi-node application clustering.

# Architecture

## Deployment Layout

- Both EC2 instances are created inside a single AWS VPC and a single public subnet.
- Each instance receives an Elastic IP.
- Internet traffic for the application enters through the App Server.
- Internet access to the Monitoring Server is limited to SSH and Grafana.
- Prometheus scrapes metrics from the App Server over the private network.
- Promtail ships logs from the App Server to Loki on the Monitoring Server over the private network.

## App Server

- Nginx reverse proxy
- React frontend container
- Node.js backend container
- MongoDB container
- Node Exporter
- Promtail
- Fail2ban

## Monitoring Server

- Prometheus
- Grafana
- Loki
- Alertmanager
- Fail2ban

## Reverse Proxy and HTTPS Flow

- Nginx listens on port `80` for the initial ACME challenge flow.
- Ansible obtains the Let's Encrypt certificate with `certbot certonly --webroot`.
- After the certificate exists, Ansible renders the final Nginx configuration with:
  - HTTP to HTTPS redirect on port `80`
  - TLS termination on port `443`
  - reverse proxy to the frontend on `localhost:3000`
  - reverse proxy to the backend on `localhost:5000`

## Monitoring and Logging Flow

- Prometheus scrapes:
  - itself on `localhost:9090`
  - Node Exporter on the App Server private IP at port `9100`
- Promtail reads:
  - `/var/log/syslog`
  - `/var/log/nginx/*.log`
- Promtail pushes logs to Loki on the Monitoring Server private IP at port `3100`
- Grafana connects to Prometheus and Loki
- Alertmanager sends notifications to Telegram

# Infrastructure Provisioning

Terraform is responsible for provisioning AWS infrastructure in [`terraform/`](./terraform):

- VPC: `10.0.0.0/24`
- Public subnet: `10.0.0.0/25`
- Internet Gateway
- Public route table
- App Server EC2
- Monitoring Server EC2
- App Server Elastic IP
- Monitoring Server Elastic IP
- Security Groups for both nodes

## Current Security Group Design

### App Server

- `22/tcp` from `my_ip`
- `80/tcp` from `0.0.0.0/0`
- `443/tcp` from `0.0.0.0/0`
- `9100/tcp` from the Monitoring Server security group only

### Monitoring Server

- `22/tcp` from `my_ip`
- `3000/tcp` from `my_ip`
- `3100/tcp` from the App Server security group only

Prometheus (`9090`) and Alertmanager (`9093`) are not opened in the AWS Security Group.

# Configuration Management

Ansible is responsible for host configuration in [`ansible/`](./ansible):

- installing common system packages
- installing Docker Engine and Docker Compose v2
- configuring UFW on both nodes
- configuring Fail2ban on both nodes
- deploying the application stack on the App Server
- deploying the monitoring stack on the Monitoring Server
- managing Nginx configuration
- obtaining and renewing Let's Encrypt certificates
- configuring Promtail, Prometheus, Alertmanager, and related templates

The main Ansible workflow is:

1. `common.yml`
2. `docker.yml`
3. `security.yml`
4. `monitoring-node.yml`
5. `app-node.yml`

# Application Deployment

The application is deployed with Docker Compose on the App Server using [`ansible/files/compose.app.yml`](./ansible/files/compose.app.yml).

## Application Services

- `mongo`
- `backend`
- `frontend`

## Current Runtime Model

- `backend` is bound to `127.0.0.1:5000`
- `frontend` is bound to `127.0.0.1:3000`
- only Nginx exposes the application publicly
- MongoDB persists data in a Docker volume on the App Server

The backend environment file is generated by Ansible from [`ansible/templates/app.env.j2`](./ansible/templates/app.env.j2).

# Monitoring and Logging

The monitoring stack is deployed with Docker Compose on the Monitoring Server using [`ansible/files/compose.monitoring.yml`](./ansible/files/compose.monitoring.yml).

## Prometheus

- configuration template: [`ansible/templates/prometheus.yml.j2`](./ansible/templates/prometheus.yml.j2)
- scrapes the App Server Node Exporter through the private IP
- uses static configuration, not service discovery

## Grafana

- runs in a container on port `3000`
- stores data in a Docker volume on the Monitoring Server

## Loki

- runs in a container on port `3100`
- receives logs from Promtail over the private network

## Promtail

- installed on the App Server as a systemd-managed binary
- configuration template: [`ansible/templates/promtail-config.yml.j2`](./ansible/templates/promtail-config.yml.j2)
- currently ships system logs and Nginx logs only

## Alertmanager

- configuration template: [`ansible/templates/alertmanager.yml.j2`](./ansible/templates/alertmanager.yml.j2)
- receives alerts from Prometheus
- forwards notifications to Telegram

## Alert Rules

- alert rules template: [`ansible/templates/alert.rules.yml.j2`](./ansible/templates/alert.rules.yml.j2)
- current repository includes a CPU usage alert based on Node Exporter metrics

# Security

## HTTPS

- HTTPS certificates are issued with Let's Encrypt Certbot
- Ansible obtains certificates with `certbot certonly --webroot`
- Nginx TLS configuration is managed by Ansible templates
- `certbot.timer` is enabled for automatic renewal

## AWS Security Groups

- App Server exposes only `22`, `80`, and `443` publicly
- Monitoring Server exposes only `22` and `3000` publicly
- Node Exporter (`9100`) and Loki (`3100`) are private-only at the security group layer

## UFW

Ansible configures UFW separately on each node:

- App Server:
  - allow `22` from `admin_ip_cidr`
  - allow `80` and `443` publicly
  - allow `9100` from the Monitoring Server private IP
- Monitoring Server:
  - allow `22` from `admin_ip_cidr`
  - allow `3000` from `admin_ip_cidr`
  - allow `3100` from the App Server private IP

## Fail2ban

- App Server:
  - `sshd`
  - `nginx-http-auth`
  - `nginx-botsearch`
- Monitoring Server:
  - `sshd`

## Private Services

The intended access model is:

- App Node:
  - public: `80`, `443`
  - private: `9100`
- Monitoring Node:
  - public: `22`, `3000`
  - private: `3100`
  - tunneled/local access: `9090`, `9093`

Note: the monitoring Docker Compose file still publishes `9090`, `3100`, and `9093` on the host. Internet exposure is reduced by AWS Security Groups and UFW rather than by loopback-only container binding.

## SSH Access

- SSH access is key-based
- inventory examples should use placeholders such as `<PATH_TO_PRIVATE_KEY>`
- helper scripts should be updated to avoid `StrictHostKeyChecking=no` in production use

# Deployment Workflow

## 1. Provision Infrastructure

Run Terraform from [`terraform/`](./terraform):

```bash
terraform init
terraform plan
terraform apply
```

Terraform outputs:

- App Server public IP
- App Server private IP
- Monitoring Server public IP
- Monitoring Server private IP

## 2. Prepare Ansible Variables

Create local variable files from the examples:

```bash
cp ansible/inventory.ini.example ansible/inventory.ini
cp ansible/group_vars/all/main.yml.example ansible/group_vars/all/main.yml
cp ansible/group_vars/all/secrets.yml.example ansible/group_vars/all/secrets.yml
```

Set generic values such as:

- `<APP_PUBLIC_IP>`
- `<MONITORING_PUBLIC_IP>`
- `<APP_PRIVATE_IP>`
- `<MONITORING_PRIVATE_IP>`
- `<PATH_TO_PRIVATE_KEY>`
- `<YOUR_DOMAIN>`
- `<CERTBOT_EMAIL>`

## 3. Ensure DNS Is Ready

Point `domain_name` to the App Server public IP before running the full Ansible deployment so Let's Encrypt validation can succeed.

## 4. Configure Servers and Deploy Services

Run:

```bash
ansible-playbook -i ansible/inventory.ini ansible/playbooks/site.yml
```

This workflow:

1. configures both EC2 instances
2. installs Docker, UFW, and Fail2ban
3. deploys the monitoring stack
4. deploys the application stack
5. configures Nginx
6. obtains the Let's Encrypt certificate
7. enables automatic certificate renewal

# Demo Scenarios

## HTTPS Verification

Verify that the application redirects to HTTPS and serves the certificate:

```bash
curl -I http://<YOUR_DOMAIN>
curl -I https://<YOUR_DOMAIN>
```

## Grafana Access

Grafana is intended to be reachable at:

```text
http://<MONITORING_IP>:3000
```

## Prometheus and Alertmanager Access

Prometheus and Alertmanager are intended to be accessed through an SSH tunnel. The repository includes a helper script:

```bash
./scripts/open-monitoring.sh <MONITORING_IP> <PATH_TO_PRIVATE_KEY>
```

This opens local tunnels for:

- `localhost:3000` → Grafana
- `localhost:9090` → Prometheus
- `localhost:9093` → Alertmanager

## Telegram Alerts

Trigger load or CPU pressure using the helper scripts in [`scripts/`](./scripts) to observe Prometheus alerts and Alertmanager notifications.

## Loki LogQL Queries

After opening Grafana, you can inspect logs from Loki using LogQL queries such as:

```logql
{job="nginx"}
```

```logql
{job="varlogs"}
```

# Current Limitations

- The App Server runs Nginx, frontend, backend, and MongoDB on the same EC2 instance.
- MongoDB is not isolated on a dedicated database host.
- The repository does not include a CI/CD pipeline.
- Terraform outputs are not automatically converted into an Ansible inventory.
- Monitoring uses static targets instead of dynamic service discovery.
- Promtail currently ships only system logs and Nginx logs, not Docker container logs.
- The monitoring Docker Compose file publishes ports broadly on the host and relies on firewall controls for isolation.
- HTTPS automation depends on DNS already pointing to the App Server before Ansible runs.
- Persistent data for MongoDB, Prometheus, and Grafana stays on the EC2 instances and is not backed up by this repository.
- The repository does not contain additional material in `docs/`.
